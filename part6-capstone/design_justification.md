## Storage Systems
For predicting patient readmission risk, the most suitable storage system is a Data Lake. Machine learning models require access to large volumes of raw, granular clinical data such as EHR records, diagnoses, lab results, and treatment timelines. A lake supports structured, semi‑structured, and unstructured formats without enforcing rigid schemas, making it ideal for historical medical data. For doctor-facing natural‑language queries, a Vector Database is essential. Doctors ask questions in plain English, and the system must retrieve semantically relevant patient notes, discharge summaries, and clinical narratives. A vector DB stores embeddings of these documents, enabling semantic search that keyword-based systems cannot achieve. For monthly management reporting, a Data Warehouse is the best choice. Warehouses excel at structured analytics, providing optimized performance for dashboards related to bed occupancy, department-wise costs, and monthly KPIs. Finally, real-time ICU vitals require a streaming storage layer, typically a combination of a message broker (Kafka/Event Hub) and a time-series or lake-based streaming sink. This ensures high-throughput ingestion and low-latency access for real-time monitoring dashboards.

## OLTP vs OLAP Boundary
The OLTP boundary begins at the Data source level and ends at the operational hospital systems: EHR systems, ICU monitoring devices, billing systems, and doctor-facing applications. These systems are optimized for fast writes, immediate consistency, and patient-critical operations. Once data leaves these systems—via batch ETL, streaming pipelines, or APIs—it enters the analytical ecosystem. The OLAP boundary begins at the ingestion layer, where data is transformed, stored, and optimized for analytics. The Data Lake, Data Warehouse, and Vector Database all operate on the OLAP side. They support workloads such as predictive modeling, semantic search, and aggregated reporting, none of which require the strict transactional guarantees of OLTP systems. This separation ensures that analytical workloads never interfere with patient‑critical operations and maintains system reliability.

## Trade-offs
A major trade-off in this architecture is the complexity introduced by using multiple specialized storage systems—a lake, warehouse, and vector database. While each system is chosen for a specific purpose, maintaining them increases operational overhead, monitoring requirements, and integration complexity. Data engineers must ensure consistent pipelines, schema alignment, and governance across systems. To mitigate this, the hospital can adopt a Lakehouse approach for partial consolidation. Modern Lakehouse platforms support ACID tables, BI-friendly schemas, and machine learning workloads within the same environment. This reduces duplication and simplifies ETL. The vector database can remain separate, but the lake and warehouse can be unified, lowering operational burden. Strong metadata management, automated pipeline orchestration, and centralized governance further reduce complexity while preserving the strengths of each component.
